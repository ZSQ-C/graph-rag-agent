
# Planner 模块代码总结

## 一、核心总览

### 核心定位
Planner 模块是 GraphRAG 多智能体系统的计划生成与管理核心，主要解决了查询理解不充分、任务分解不合理、执行计划不完善的问题。它通过 Clarifier 分析查询是否需要澄清并收集必要的上下文信息；通过 TaskDecomposer 将复杂查询分解为一系列有序的、可执行的子任务；通过 PlanReviewer 对生成的计划进行审校和验证，确保计划的合理性和可执行性；通过 BasePlanner 作为统一入口，协调各个子模块的工作并确保反思任务被自动附加到计划中。该模块适用于需要将复杂查询分解为系统化执行流程的场景，以及需要确保查询理解准确性的场景。

### 整体流程串讲
当需要为用户查询生成执行计划时，首先调用 BasePlanner 的 generate_plan() 方法，该方法作为整个 Planner 模块的统一入口。generate_plan() 首先调用 _ensure_plan_context() 确保状态对象中包含必要的计划上下文信息。接着，调用 Clarifier 的 analyze() 方法分析查询是否需要澄清：如果需要澄清，Clarifier 会生成澄清问题并通过 _record_clarification() 记录到状态中；如果不需要澄清或澄清已完成，则继续下一步。然后，调用 TaskDecomposer 的 decompose() 方法，将查询分解为一系列子任务，并构建任务依赖关系图。任务分解完成后，调用 _ensure_reflection_task() 确保在计划的末尾自动附加一个反思任务，用于评估整个执行过程的质量并提出改进建议。之后，调用 PlanReviewer 的 review() 方法对生成的计划进行审校和验证，检查任务依赖关系、任务可行性、任务完整性等方面，如果发现问题则要求 TaskDecomposer 重新生成计划。最后，将经过审校的完整计划返回给调用者。整个流程中，BasePlanner 负责协调各个子模块的调用顺序和数据流转，Clarifier 负责查询澄清，TaskDecomposer 负责任务分解，PlanReviewer 负责计划审校，四者配合确保生成一个高质量、可执行的任务计划。

## 二、模块拆分

### 初始化模块
该模块的作用是提供 Planner 模块的公共接口和基础组件定义，位于整个 Planner 模块的最顶层，是外部调用者与模块交互的入口。它主要通过 __init__.py 文件实现，导出 BasePlanner、Clarifier、TaskDecomposer、PlanReviewer 等核心类，以及相关的数据模型，使外部调用者可以方便地导入和使用这些组件。

### 核心入口方法模块
该模块的作用是协调各个子模块完成计划生成的完整流程，位于 Planner 模块的核心位置，是各个子模块的协调者。它主要通过 BasePlanner 类的 generate_plan() 方法实现，负责调用 Clarifier、TaskDecomposer、PlanReviewer 等子模块，并确保反思任务被自动附加到计划中，为外部调用者提供一个统一、简洁的计划生成接口。

### 分支逻辑方法模块
该模块的作用是实现计划生成过程中的各个分支逻辑，位于核心入口方法模块的下层，与核心入口方法紧密配合。它包括 Clarifier 的 analyze() 方法（处理查询澄清分支）、TaskDecomposer 的 decompose() 方法（处理任务分解分支）、PlanReviewer 的 review() 方法（处理计划审校分支），以及 BasePlanner 中的 _ensure_reflection_task()、_ensure_plan_context()、_record_clarification() 等辅助方法，主要负责实现计划生成过程中的各个具体分支逻辑。

### 具体实现方法模块
该模块的作用是实现各个分支逻辑的具体细节，位于分支逻辑方法模块的下层，是分支逻辑的具体实现者。它包括 Clarifier 的 _invoke_llm()、_parse_response() 方法，TaskDecomposer 的 _build_task_graph() 方法，PlanReviewer 的 _resolve_task_graph() 方法等，主要负责 LLM 调用、响应解析、任务图构建、任务图解析等具体实现细节。

### 辅助方法模块
该模块的作用是为核心流程和分支逻辑提供支持性功能，位于各个实现模块的下层，为其他模块提供通用的辅助功能。它包括各个数据模型的定义（ClarificationResult、TaskDecompositionResult、PlanReviewOutcome、PlanValidationResult 等），以及各种通用的辅助方法，主要负责数据结构定义、数据验证、通用工具函数等辅助性功能。

## 三、方法详细解析

### BasePlanner.generate_plan()

#### 方法文字流程串讲
generate_plan() 是 BasePlanner 的核心入口方法，负责协调各个子模块完成计划生成的完整流程。方法开始时，首先调用 _ensure_plan_context() 确保状态对象中包含必要的计划上下文信息，初始化计划相关的字段。接着，调用 Clarifier 的 analyze() 方法分析查询是否需要澄清：如果返回的 ClarificationResult 中 needs_clarification 为 True，则说明需要澄清，此时调用 _record_clarification() 将澄清问题和相关信息记录到状态中，并返回一个包含澄清请求的计划；如果 needs_clarification 为 False，则说明查询足够清晰，可以继续进行任务分解。然后，调用 TaskDecomposer 的 decompose() 方法，将查询分解为一系列子任务，并构建任务依赖关系图，得到 TaskDecompositionResult。之后，调用 _ensure_reflection_task() 确保在任务列表的末尾自动附加一个反思任务，用于评估整个执行过程的质量。反思任务附加完成后，调用 PlanReviewer 的 review() 方法对生成的计划进行审校和验证，得到 PlanReviewOutcome：如果 PlanReviewOutcome 中的 is_valid 为 True，则说明计划通过审校，可以返回；如果 is_valid 为 False，则说明计划存在问题，需要根据 feedback 重新生成计划，此时可以循环调用 TaskDecomposer 和 PlanReviewer 直到计划通过审校或达到最大重试次数。最后，将经过审校的完整计划返回给调用者。

#### 强制 5 要素
- **入参**：
  - state: Any（必填），包含查询信息和上下文的状态对象
- **核心逻辑**：确保计划上下文 → 查询澄清 → 任务分解 → 附加反思任务 → 计划审校 → 返回计划
- **输出形式**：返回生成的执行计划对象
- **底层关键依赖**：_ensure_plan_context()、Clarifier.analyze()、_record_clarification()、TaskDecomposer.decompose()、_ensure_reflection_task()、PlanReviewer.review()
- **关键代码片段**：
```python
def generate_plan(self, state: Any) -&gt; Any:
    state = self._ensure_plan_context(state)
    
    clarification_result = self.clarifier.analyze(state)
    if clarification_result.needs_clarification:
        state = self._record_clarification(state, clarification_result)
        return self._build_clarification_plan(state, clarification_result)
    
    decomposition_result = self.task_decomposer.decompose(state)
    decomposition_result = self._ensure_reflection_task(decomposition_result)
    
    review_outcome = self.plan_reviewer.review(decomposition_result, state)
    if not review_outcome.is_valid:
        return self._handle_review_feedback(state, review_outcome)
    
    return self._build_final_plan(state, decomposition_result)
```

### BasePlanner._ensure_reflection_task()

#### 方法文字流程串讲
_ensure_reflection_task() 是一个辅助方法，用于确保任务列表中包含反思任务。方法开始时，首先检查任务列表的最后一个任务是否已经是反思任务：如果是，则直接返回原有的 TaskDecompositionResult，不做任何修改；如果不是，则在任务列表的末尾添加一个新的反思任务。反思任务的类型通常标记为 "reflection"，它的依赖关系设置为依赖于前面所有其他任务的完成，确保反思在所有执行任务完成后才进行。最后，返回更新后的 TaskDecompositionResult。这个方法确保了每个计划都包含反思环节，用于评估执行过程的质量并提出改进建议。

#### 强制 5 要素
- **入参**：
  - decomposition_result: TaskDecompositionResult（必填），任务分解结果对象
- **核心逻辑**：检查是否已有反思任务 → 如没有则添加反思任务 → 设置反思任务的依赖关系
- **输出形式**：返回更新后的 TaskDecompositionResult
- **底层关键依赖**：TaskDecompositionResult 类、Task 类（假设）
- **关键代码片段**：
```python
def _ensure_reflection_task(self, decomposition_result: TaskDecompositionResult) -&gt; TaskDecompositionResult:
    tasks = decomposition_result.tasks
    if tasks and tasks[-1].task_type == "reflection":
        return decomposition_result
    
    reflection_task = Task(
        task_id=f"task_{len(tasks) + 1}",
        task_type="reflection",
        description="反思整个执行过程，评估答案质量并提出改进建议",
        dependencies=[task.task_id for task in tasks]
    )
    
    tasks.append(reflection_task)
    return decomposition_result
```

### Clarifier.analyze()

#### 方法文字流程串讲
analyze() 是 Clarifier 的核心方法，用于分析查询是否需要澄清。方法开始时，首先构建 LLM 调用的提示词，将查询文本和上下文信息嵌入到提示词中。接着，调用 _invoke_llm() 方法将提示词发送给 LLM，获取 LLM 的响应。然后，调用 _parse_response() 方法解析 LLM 的响应，提取出是否需要澄清（needs_clarification）、澄清问题列表（clarification_questions）、澄清理由（reasoning）等信息，并将这些信息封装到 ClarificationResult 对象中。最后，返回 ClarificationResult 对象。整个过程中，如果 LLM 调用失败或响应解析失败，方法会记录错误日志并返回一个默认的 ClarificationResult（needs_clarification=False），确保系统的健壮性。

#### 强制 5 要素
- **入参**：
  - state: Any（必填），包含查询信息和上下文的状态对象
- **核心逻辑**：构建提示词 → 调用 LLM → 解析响应 → 封装结果
- **输出形式**：返回 ClarificationResult 对象
- **底层关键依赖**：_invoke_llm()、_parse_response()、ClarificationResult 类
- **关键代码片段**：
```python
def analyze(self, state: Any) -&gt; ClarificationResult:
    try:
        prompt = self._build_prompt(state)
        llm_response = self._invoke_llm(prompt)
        return self._parse_response(llm_response)
    except Exception as e:
        logger.error(f"Clarification analysis failed: {str(e)}")
        return ClarificationResult(
            needs_clarification=False,
            clarification_questions=[],
            reasoning="Clarification analysis failed, proceeding without clarification"
        )
```

### TaskDecomposer.decompose()

#### 方法文字流程串讲
decompose() 是 TaskDecomposer 的核心方法，用于将查询分解为一系列子任务。方法开始时，首先构建 LLM 调用的提示词，将查询文本、上下文信息、以及任务分解的要求嵌入到提示词中。接着，调用 LLM 获取任务分解的响应。然后，解析 LLM 的响应，提取出各个子任务的信息，包括任务 ID、任务类型、任务描述、依赖关系等。之后，调用 _build_task_graph() 方法，基于提取出的子任务信息构建任务依赖关系图，确保任务的执行顺序正确。最后，将子任务列表和任务依赖关系图封装到 TaskDecompositionResult 对象中并返回。整个过程中，如果任何步骤出现异常，方法会记录错误日志并返回一个包含默认任务的 TaskDecompositionResult，确保系统的健壮性。

#### 强制 5 要素
- **入参**：
  - state: Any（必填），包含查询信息和上下文的状态对象
- **核心逻辑**：构建提示词 → 调用 LLM → 解析响应 → 构建任务图 → 封装结果
- **输出形式**：返回 TaskDecompositionResult 对象
- **底层关键依赖**：_build_task_graph()、TaskDecompositionResult 类
- **关键代码片段**：
```python
def decompose(self, state: Any) -&gt; TaskDecompositionResult:
    try:
        prompt = self._build_prompt(state)
        llm_response = self._invoke_llm(prompt)
        raw_tasks = self._parse_tasks(llm_response)
        task_graph = self._build_task_graph(raw_tasks)
        return TaskDecompositionResult(
            tasks=raw_tasks,
            task_graph=task_graph,
            reasoning="Task decomposition completed successfully"
        )
    except Exception as e:
        logger.error(f"Task decomposition failed: {str(e)}")
        return self._build_fallback_decomposition(state)
```

### PlanReviewer.review()

#### 方法文字流程串讲
review() 是 PlanReviewer 的核心方法，用于对生成的计划进行审校和验证。方法开始时，首先调用 _resolve_task_graph() 方法解析任务依赖关系图，检查任务依赖关系是否合理、是否存在循环依赖、是否存在孤立任务等问题。接着，构建 LLM 调用的提示词，将任务列表、任务依赖关系图、查询信息等嵌入到提示词中，要求 LLM 对计划进行全面审校。然后，调用 LLM 获取审校响应。之后，解析 LLM 的响应，提取出计划是否有效（is_valid）、验证结果列表（validation_results）、审校反馈（feedback）等信息。最后，将这些信息封装到 PlanReviewOutcome 对象中并返回。整个过程中，除了 LLM 审校外，还会进行一些规则性检查，例如任务 ID 唯一性检查、依赖关系有效性检查等，确保计划的基本质量。

#### 强制 5 要素
- **入参**：
  - decomposition_result: TaskDecompositionResult（必填），任务分解结果对象
  - state: Any（必填），包含查询信息和上下文的状态对象
- **核心逻辑**：解析任务图 → 规则性检查 → LLM 审校 → 解析响应 → 封装结果
- **输出形式**：返回 PlanReviewOutcome 对象
- **底层关键依赖**：_resolve_task_graph()、PlanReviewOutcome 类、PlanValidationResult 类
- **关键代码片段**：
```python
def review(self, decomposition_result: TaskDecompositionResult, state: Any) -&gt; PlanReviewOutcome:
    validation_results = []
    
    task_graph_validation = self._resolve_task_graph(decomposition_result.task_graph)
    validation_results.append(task_graph_validation)
    
    rule_validations = self._perform_rule_checks(decomposition_result)
    validation_results.extend(rule_validations)
    
    llm_review = self._invoke_llm_review(decomposition_result, state)
    validation_results.append(llm_review)
    
    is_valid = all(v.is_valid for v in validation_results)
    feedback = "\n".join([v.message for v in validation_results if not v.is_valid])
    
    return PlanReviewOutcome(
        is_valid=is_valid,
        validation_results=validation_results,
        feedback=feedback
    )
```
